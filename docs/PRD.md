# Dooeyflow (요식업 재고관리 앱) 요구사항 정의서 (PRD)
*버전: v0.3*

## 1. 프로젝트 개요
- **목적**: 요식업 매장 사장님들을 위한 실시간 및 일괄 재고 관리 플랫폼
- **핵심 가치**: 토스 플레이스 연동을 통한 자동 재고 차감, 레시피 연동 원자재 정밀 관리, 레시피 매뉴얼화, 재고 부족 앱 푸시 알림
- **타겟 유저**: 중소규모 식당/카페 사장님

### 1.1. 지원 플랫폼 (Platform Targets)
사장님은 PC 데스크톱과 모바일 기기를 오가며 매장을 관리하므로, **웹과 네이티브 앱을 모두 제공**한다.

| 채널 | 대상 | 비고 |
|------|------|------|
| **웹 (반응형)** | 데스크톱 + 모바일 브라우저 | 1차 출시 범위. Next.js 반응형 |
| **iPhone 앱** | iOS 16.4 이상 | Capacitor 래핑, App Store 배포 |
| **iPad 앱** | iPadOS 16.4 이상 | iPhone 앱과 동일 코드의 Universal 앱 |

- **단일 코드베이스 원칙**: Next.js 웹앱을 `Capacitor`로 네이티브 쉘에 래핑하여 iPhone/iPad 앱을 빌드한다. 웹과 앱이 동일한 프론트엔드 코드를 공유하므로 유지보수 비용을 최소화한다.
- **반응형 우선**: 모든 화면은 데스크톱·태블릿·모바일 폭에서 깨지지 않도록 설계한다. iPad는 태블릿 레이아웃(2단 구성 등)을 별도 고려한다.

## 2. 핵심 기능 요구사항

### 2.1. 온보딩 및 POS 연동
- **회원가입 및 매장 생성**: 사장님 계정 생성 및 관리할 매장 정보 등록
- **토스 플레이스 OpenAPI 연동 (선택형)**
  - 토스 POS를 사용하는 매장의 경우, API 연동에 필요한 정보 입력
  - **수신 방식 (투트랙 검증)**: 
    1. **웹훅(Webhook)**: 실시간 주문 발생 시 즉각적인 재고 차감 알림 수신
    2. **폴링(Polling)**: 네트워크 단절 등으로 웹훅이 유실될 경우를 대비한 주기적 데이터 정합성 교차 검증
  - 토스 POS 미사용 시 '수동 관리 모드'로 동작

### 2.2. 메뉴, 식자재, 레시피 관리 (BOM & 매뉴얼)
- **원자재/식자재 등록**: 매장에서 사용하는 원자재 리스트 등록 (단위 포함)
- **레시피 등록 (복합 기능)**:
  - **재고 차감용 (BOM)**: 메뉴 판매 시 소모되는 원자재와 정확한 수량(예: 우유 150ml, 원두 18g 등)을 매핑. 중간 단계의 '반제품'을 생성하지 않고, 대용량 식자재에서 비율에 맞게 직접 정밀하게 차감되도록 직관적으로 설계.
  - **조리 매뉴얼용**: 알바생이나 직원이 참고할 수 있는 조리법(사진, 텍스트 가이드) 포함.
  - **확장성**: 추후 유저 간 훌륭한 레시피를 포인트로 사고파는 커뮤니티/마켓 기능 염두.

### 2.3. 판매 기록 및 재고 차감
- **자동 차감 모드 (토스 연동 매장)**
  - 토스 웹훅 및 주기적 API 조회를 통해 판매된 메뉴 식별
  - 메뉴에 연결된 레시피(BOM)를 바탕으로 등록된 원자재 재고 자동 마이너스(-) 처리
- **수동 차감 모드 (미연동 매장)**
  - **일괄 입력 방식**: 장사 마감 후 혹은 특정 시간에 사장님이 '하루 판매량'을 메뉴별로 일괄 입력
  - 입력 완료 시 해당 수량만큼 레시피를 곱하여 원자재 일괄 차감

### 2.4. 재고 모니터링 및 알림
- **안전 재고 임계값 설정**: 품목별 최소 수치(안전 재고) 설정
- **자동 알림 발송**: 재고 차감 후 남은 수량이 설정 수치 미만으로 떨어질 경우 **앱 푸시(App Push)** 알림 발송
  - **네이티브 푸시 (iPhone/iPad)**: Capacitor 앱은 **APNs(Apple Push Notification service)**를 통해 백그라운드에서도 신뢰성 있게 알림 수신. 이것이 푸시 알림의 1차 채널.
  - **웹 푸시 (브라우저)**: 웹 사용자에게는 Web Push(브라우저 알림)로 보조 발송. 단, iOS Safari 웹 푸시는 PWA 설치 + iOS 16.4+ 제약이 있어 앱 설치를 권장한다.
  - **알림 채널 분기**: 사용자가 앱/웹 중 어디서 로그인했는지에 따라 디바이스 토큰을 구분 저장하고 적절한 채널로 발송.

## 3. 향후 확장 (Phase 2 - Out of Scope)
- **레시피 마켓**: 점주들끼리 레시피를 포인트로 구매/판매하는 플랫폼 기능
- **공급처 관리 및 발주 연동**: 식자재 유통업체 연동, 원클릭 자동 발주
- **다중 매장 관리**: 프랜차이즈 본사 및 다점포 사장님을 위한 통합 관리 및 권한 통제

## 4. 비기능 요구사항 (Non-Functional Requirements)

### 4.1. 인증/인가
- **계정 방식**: 사장님 계정은 **이메일+비밀번호를 기본**으로 하고, 모바일 편의를 위해 **소셜 로그인(카카오, 네이버, 애플)**을 확장 제공한다.
  - **애플 로그인 필수 사유**: App Store 심사 규정(Guideline 4.8)상 iOS 앱이 서드파티 소셜로그인(카카오/네이버)을 제공하면 'Sign in with Apple'을 함께 제공해야 한다. 따라서 iPhone/iPad 앱에는 애플 로그인을 반드시 포함한다.
- **세션**: 웹과 앱이 동일 API를 공유하므로 JWT 기반 토큰 인증을 사용하고, 앱은 토큰을 OS 보안 저장소(iOS Keychain)에 저장한다.
- **매장 격리**: 모든 API는 로그인 사용자의 `store_id` 범위로만 데이터에 접근(Row-Level 격리).

### 4.2. 오프라인/네트워크 내성
- 매장 현장은 네트워크가 불안정할 수 있으므로, **수동 일괄 입력 화면은 입력 중 네트워크가 끊겨도 데이터가 유실되지 않도록** 로컬 임시 저장 후 재전송(재시도)을 지원한다.
- 토스 폴링은 웹훅 유실 시 정합성을 복구하는 안전망 역할을 한다.

### 4.3. 보안
- **토스 웹훅 서명 검증**: 수신한 웹훅이 토스에서 온 것인지 서명(Signature)을 반드시 검증한 뒤 처리한다.
- **민감정보**: 토스 API 키 등은 환경변수/시크릿으로 관리하고 절대 코드/로그에 노출하지 않는다.

### 4.4. 성능
- 토스 웹훅 수신 → 재고 차감 응답은 비동기로 빠르게 ACK 처리(이중 차감 멱등성 보장).
- 모바일 앱/웹 모두 주요 화면(대시보드, 재고 목록) 초기 로딩 체감 속도를 고려한다.

## 5. 해결해야 할 주요 기술/비즈니스 과제
1. **재고 수량의 정밀도**: 소수점 계산 오류(부동소수점 이슈) 방지를 위해 DB 단에서 `DECIMAL` 타입 사용.
2. **이중 차감 방지**: 웹훅과 폴링이 겹칠 때 동일 주문으로 재고가 두 번 깎이지 않도록 멱등성(Idempotency) 보장 키 설계.
3. **푸시 알림 신뢰성**: iOS 웹 푸시 제약을 우회하기 위해 네이티브 앱(APNs)을 푸시 1차 채널로 삼고, 디바이스 토큰 관리/만료 처리를 설계.
4. **단일 코드베이스 유지**: 웹과 iPhone/iPad 앱이 Capacitor를 통해 같은 코드를 공유하되, 네이티브 기능(푸시, 보안 저장소)은 플랫폼 분기 처리.
